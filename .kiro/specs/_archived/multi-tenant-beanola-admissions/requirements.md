# Requirements Document

## Introduction

This document defines requirements for converting the original single-tenant MIHAS/KATC admissions system into a **Beanola-owned multi-school admissions platform**. Beanola is the platform owner; MIHAS, KATC, and future schools are client institutions. Students apply **program-first**: they choose a canonical program and intake, the backend assigns an eligible school-specific offering, the student reviews the assigned school + fee + required documents before paying, Beanola collects payment centrally, payments are tagged by institution for settlement, and official documents are generated by the backend using the assigned school's brand and versioned templates/assets.

A substantial foundation already exists in the repository (see `docs/multi-tenant-beanola-handover.md` and `docs/multi-tenant-beanola-progress.md`): the additive SQL migration, tenant models, `OfferingAssignmentService`, `InstitutionContextService`, `AccessScopeService`, `DocumentTemplateService`, the canonical catalog endpoints, tenant admin APIs, official PDF generation, and the program-first wizard. **This spec does not assume that foundation is correct, complete, or production-safe.** Its primary mission is to *prove the canonical-ID model and the access-scope model under real database conditions*, harden every cross-tenant boundary, and only then complete the UI/UX and operational surfaces.

The single overriding invariant is: **no non-super-admin actor may ever read, infer, export, or act on another school's data** — not through lists, counts, empty states, search, exports, document URLs, payment receipts, settlement summaries, analytics, timing, or error messages. The second invariant is: **canonical IDs are the source of truth; legacy string fields (`applications.institution`, `applications.program`, `applications.intake`) are immutable display snapshots only** and must never drive new business logic.

The scope is additive and backward-compatible. Existing `/api/v1/...` routes, the `{"success": true, "data": ...}` envelope, cookie-based auth, CSRF protection, admissions auto-save, the Lenco mobile-money-first payment UX, and all existing production records must remain usable. Legacy string columns are retained. No arbitrary DOCX/PDF merge engines are introduced in V1.

## Glossary

- **Beanola**: The platform owner and central payment collector. Default brand on the shared portal before a school is assigned.
- **Institution / School**: A client tenant (MIHAS, KATC, future schools). Row in the `institutions` table. Identity is `institutions.id`.
- **Canonical_Program**: A shared, cross-school program definition. Row in `canonical_programs`. Identity is `canonical_programs.id`. This is what a student chooses.
- **Offering**: A school-specific program. Row in the existing `programs` table, linked to a `Canonical_Program` via `programs.canonical_program_id` and owned by an institution via `programs.institution_id`.
- **Intake**: An admission window. Row in `intakes`. Per-offering availability/capacity/priority lives in `program_intakes`.
- **Program_Intake**: Row in `program_intakes` controlling offering availability, capacity, `is_active`, `assignment_priority`, and `residency_rules` for a specific offering+intake.
- **Assignment**: The backend act of selecting one eligible Offering (and its Institution) for a chosen Canonical_Program + Intake, given residency/country/nationality and optional white-label institution filter.
- **Assignment_Service**: `backend/apps/catalog/services.py:OfferingAssignmentService`.
- **Institution_Context_Service**: `backend/apps/catalog/services.py:InstitutionContextService` — resolves shared-Beanola vs white-label-school context from the request host.
- **Access_Scope_Service**: `backend/apps/catalog/services.py:AccessScopeService` — computes the set of institution/offering/application IDs a non-super-admin may access.
- **Document_Template_Service**: `backend/apps/catalog/services.py:DocumentTemplateService` — resolves safe section/token templates and versioned assets for official PDFs.
- **Canonical_IDs**: The four nullable application columns `applications.institution_id`, `applications.program_id` (canonical), `applications.program_offering_id`, `applications.intake_id`.
- **Legacy_Snapshot_Fields**: The pre-existing string columns `applications.institution`, `applications.program`, `applications.intake`. Immutable display snapshots; never written by new business logic except backfill/migration.
- **Shared_Portal**: The Beanola-branded entry point used when the request host is not a recognised white-label school domain.
- **White_Label_Portal**: A school-branded entry point resolved from an active row in `institution_domains` whose institution is active.
- **Membership**: Row in `user_institution_memberships` granting a staff user scoped access to one institution.
- **Access_Grant**: Row in `access_grants` granting explicit extra access at institution, program-offering, or application scope, optionally time-bounded via expiry.
- **Super_Admin**: A global actor with platform-wide access (per `backend/apps/accounts/permissions.py`).
- **School_Staff**: A non-super-admin admin/reviewer whose access is bounded by Memberships and Access_Grants.
- **Scope_Filters**: The dataclass returned by `Access_Scope_Service` describing whether the caller is global and which institution/offering/application IDs they may access.
- **Institution_Asset**: Versioned logo/signature/seal in `institution_assets` (R2/S3 key, MIME, checksum, version, metadata).
- **Document_Template**: Versioned safe-section/token template in `institution_document_templates` for official PDFs.
- **Required_Document**: Configurable document requirement in `institution_required_documents`, scoped per institution, offering, canonical program, or default.
- **Official_Document**: A backend-generated, canonical PDF (application slip, acceptance letter, conditional offer, finance receipt, payment receipt) that snapshots template version + asset IDs into `ApplicationDocument.verification_notes.official_document`.
- **Settlement_Metadata**: The institution/canonical-program/offering/intake snapshot written into `payments.metadata` at initiation, used for tagging and settlement reporting independent of later renames.
- **Migration_History_Prerequisite**: The `2026_05_22_migration_history_extend.sql` migration that must be applied before `apply_sql_migrations --dry-run` will run in this checkout.
- **Tenant_Migration**: `backend/scripts/2026_06_08_01_multi_tenant_beanola_admissions.sql` — the additive, idempotent multi-tenant schema migration (relocated from the runner-excluded `scripts/migrations/` directory to the top-level deployable path).

## Requirements

### Requirement 1: Canonical-ID Source of Truth

**User Story:** As a platform engineer, I want all new tenant routing and application logic to read canonical IDs rather than legacy strings, so that school/program renames and historical spelling variants never corrupt routing, reporting, or access control.

#### Acceptance Criteria

1. WHEN a new application is created through the program-first flow, THE system SHALL persist all four Canonical_IDs (`institution_id`, `program_id`, `program_offering_id`, `intake_id`) on the `applications` row in the same transaction as creation.
2. THE system SHALL treat `canonical_programs.id`, `programs.id` (as `program_offering_id`), `institutions.id`, and `intakes.id` as the sole sources of truth for routing, scoping, payment tagging, and document generation.
3. THE system SHALL NOT introduce new business logic that matches on `applications.institution`, `applications.program`, or `applications.intake` strings, except in explicit migration/backfill code or a clearly labelled legacy-fallback branch for rows with null Canonical_IDs.
4. WHILE a legacy application has one or more null Canonical_IDs, THE read paths SHALL still display that application using its Legacy_Snapshot_Fields without raising an error.
5. THE Legacy_Snapshot_Fields SHALL remain populated and SHALL NOT be dropped, overwritten, or repurposed by any new flow.
6. WHERE a new domain concept (canonical program, offering, institution, intake, membership, grant) is added, THE concept SHALL be registered in `docs/canonical-truth-map.md` before merge, and a drift-guard test SHALL be required for any concept that has a frontend mirror.

### Requirement 2: Offering Assignment Correctness

**User Story:** As a student, I want the platform to deterministically assign me to an eligible school for my chosen program and intake, so that I always see a single correct school, fee, and document list before I pay.

#### Acceptance Criteria

1. WHEN `Assignment_Service.assign(program_id, intake_id, country, nationality, institution_id?)` is called, THE Assignment_Service SHALL consider only Offerings where `canonical_program_id` matches, the Offering is active (`is_active=true` AND `offering_status='active'`), the Offering is linked to the requested intake via `program_intakes`, and the matching `program_intakes` row is active or legacy-null-active.
2. WHERE an `institution_id` (white-label context) is supplied, THE Assignment_Service SHALL restrict candidates to that institution only.
3. WHEN evaluating each candidate, THE Assignment_Service SHALL apply the Offering's `assignment_rules` and the Program_Intake's `residency_rules` against the supplied `country` and `nationality`, excluding candidates whose rules block the applicant.
4. WHEN capacity for a candidate Program_Intake is exhausted, THE Assignment_Service SHALL exclude that candidate unless a documented waitlist path applies.
5. WHEN multiple candidates remain, THE Assignment_Service SHALL sort by Program_Intake `assignment_priority` first, then Offering `assignment_priority` (lower wins), then a deterministic tie-break on Offering code/id, and SHALL return exactly one Offering, Institution, Canonical_Program, Intake, and the resolved Required_Documents.
6. IF no candidate is eligible, THEN THE Assignment_Service SHALL raise an error carrying stable code `NO_ELIGIBLE_OFFERING` and the API SHALL return a recoverable response with user-facing guidance (choose another intake, contact admissions, or join an interest list).
7. WHEN a student submits an application, THE submission path SHALL re-run assignment validation (eligibility and capacity) and SHALL NOT rely solely on the assignment captured at draft-creation time.
8. WHEN assignment succeeds or fails, THE Assignment_Service SHALL emit an Audit_Event recording the decision inputs (canonical program, intake, residency inputs, white-label institution if any) and the chosen Offering or the failure code.

### Requirement 3: White-Label and Shared Portal Context

**User Story:** As a student arriving on either the Beanola portal or a school's white-label domain, I want the brand and program choices to match the entry point, so that the experience is coherent and never leaks an unrelated school.

#### Acceptance Criteria

1. WHEN `Institution_Context_Service` resolves a request whose host matches an active `institution_domains.hostname` for an active institution, THE system SHALL return White_Label_Portal context with that institution's brand and SHALL filter catalog/offering choices to that institution.
2. WHEN the request host does not match any active white-label domain, THE system SHALL return Shared_Portal context with the Beanola brand and SHALL NOT pre-filter to any single school.
3. WHERE a white-label domain row exists but its institution is inactive or the domain is inactive, THE system SHALL fall back to a safe state (Shared_Portal Beanola brand or an explicit "unavailable" state) and SHALL NOT expose the inactive school's data.
4. WHEN matching a host, THE Institution_Context_Service SHALL match case-insensitively and SHALL ignore port suffixes.
5. IF two active `institution_domains` rows share the same hostname, THEN THE system SHALL treat this as a configuration error, resolve deterministically (or fail safe to Shared_Portal), and surface the collision to operators rather than silently picking one school's data.
6. WHILE on the Shared_Portal before assignment, THE catalog UI SHALL present canonical programs without favouring or naming any single school.

### Requirement 4: Scoped Access for School Staff (Cross-Tenant Isolation)

**User Story:** As a school staff member, I want to see and act on only my school's data, so that no other school's applicants, payments, or documents are ever exposed to me.

#### Acceptance Criteria

1. WHEN a Super_Admin makes any request, THE Access_Scope_Service SHALL grant global access.
2. WHEN a School_Staff user makes a request, THE Access_Scope_Service SHALL compute Scope_Filters from active Memberships (institution scope) and active, non-expired Access_Grants (institution, offering, or application scope) only.
3. THE application list, application review/retrieval, application exports, payments list, payment receipt, payment verify, document retrieval/download/delete, admin document upload, admin dashboard aggregates, dashboard recent activity, dashboard needs-attention counts, and admin user listing SHALL all filter results to the caller's Scope_Filters for School_Staff.
4. WHEN a School_Staff user requests a record outside their scope, THE system SHALL respond identically to a not-found case (same status, shape, and message) so that existence cannot be inferred, and SHALL NOT leak the other school's data in any field or error.
5. THE analytics endpoints, any cross-school search/count endpoints, risk views, operational reports, and any non-obvious document signed-URL paths SHALL apply Scope_Filters or be restricted to Super_Admin before exposure to School_Staff.
6. WHEN a School_Staff user has no Memberships and no Access_Grants, THE scoped surfaces SHALL return empty results scoped to "no school access" and SHALL NOT return global zero/aggregate counts that could imply platform-wide totals.
7. WHERE an Access_Grant is application-scoped, THE grant SHALL NOT confer institution-wide access; WHERE it is offering-scoped, THE grant SHALL NOT expose unrelated offerings.
8. WHEN an Access_Grant's expiry has passed, THE Access_Scope_Service SHALL exclude it from the computed scope.
9. THE production access model SHALL be membership/grant-driven; the existing test-settings-only legacy-admin compatibility path SHALL NOT be relied upon or extended into production behaviour.

### Requirement 5: Tenant Onboarding and Configuration APIs

**User Story:** As a Super_Admin operator, I want to onboard and configure schools end to end, so that a new institution can receive applications with correct branding, offerings, routing, documents, and staff access.

#### Acceptance Criteria

1. THE system SHALL expose Super_Admin tenant management endpoints under `/api/v1/admin/` for institutions, institution domains, institution assets (including multipart upload), document templates, required documents, memberships, and access grants, supporting list/create and detail PATCH (deactivate) lifecycle.
2. WHEN creating or updating an institution, domain, or offering identity, THE serializers SHALL reject slug, code, and hostname collisions with a descriptive validation error.
3. WHEN uploading an Institution_Asset, THE upload endpoint SHALL validate MIME and magic bytes for PNG/JPEG/WebP/SVG, capture a SHA-256 checksum, store a versioned R2/S3-backed row, and SHALL reject mismatched or oversized files with a clear error.
4. WHEN a new version of a logo/signature/seal is uploaded, THE system SHALL create a new versioned Institution_Asset row and SHALL NOT alter assets already referenced by previously generated Official_Documents.
5. THE access grants API SHALL support filtering by institution so a tenant page shows only that school's grants and SHALL NOT expose grants for unrelated institutions.
6. WHEN a School_Staff user calls any tenant management endpoint outside their scope, THE system SHALL deny access without leaking the existence or configuration of other schools.
7. THE Document_Template create/update API SHALL accept only safe sections and tokens and SHALL NOT accept or execute arbitrary uploaded DOCX/PDF merge documents.

### Requirement 6: Official Document Generation Integrity

**User Story:** As a school, I want official documents to render with my brand and to remain stable after I change templates or assets, so that issued documents are trustworthy and reproducible.

#### Acceptance Criteria

1. WHEN the backend generates an Official_Document (application slip, acceptance letter, conditional offer, finance receipt, payment receipt), THE renderer SHALL use the assigned school's context (brand, contacts, templates, active latest drawable logo/signature assets) where available.
2. WHEN an Official_Document is generated, THE system SHALL snapshot the template ID + version and the asset IDs used into `ApplicationDocument.verification_notes.official_document`.
3. THE backend-generated Official_Document SHALL be canonical; frontend PDF rendering SHALL be treated as preview/download helper only and SHALL NOT be presented as the official record.
4. WHEN a Document_Template renders tokens, THE renderer SHALL HTML-escape token values and SHALL restrict tokens to an allowlist, rejecting or safely ignoring unknown or injected tokens.
5. WHERE a template is missing for a document type, THE renderer SHALL fall back to a safe default body rather than failing the request.
6. WHEN template/assets change after an Official_Document was generated, THE system SHALL NOT silently regenerate or alter the previously generated document.
7. IF an SVG asset cannot be safely rendered, THEN THE renderer SHALL either render it safely or treat the type as unsupported with clear UI copy, and SHALL NOT execute untrusted SVG content.
8. WHEN an Official_Document render fails, THE system SHALL record the failure in an auditable form and SHALL surface a recoverable error to the operator.

### Requirement 7: Central Payment Collection and Settlement Tagging

**User Story:** As Beanola finance, I want every payment tagged with the institution, canonical program, offering, and intake at the time of collection, so that settlement and reporting stay correct even after schools or programs are renamed or archived.

#### Acceptance Criteria

1. WHEN a payment is initiated through any initiation path (card initiate, mobile money, defer), THE system SHALL write Settlement_Metadata into `payments.metadata` including the Beanola collector marker, institution ID/code/name, canonical program ID/name, program offering ID/code, and intake ID/name.
2. THE settlement and reporting paths SHALL derive labels from the payment Settlement_Metadata snapshot and SHALL NOT recompute historical labels solely from current institution/program names.
3. WHEN a payment receipt is produced, THE receipt SHALL remain valid and correctly attributed even if the school's branding or name later changes.
4. THE settlement summary endpoint SHALL be tenant-scoped for School_Staff and SHALL group payments by institution/offering/currency, treating missing metadata as an explicit "Unassigned" bucket without leaking other schools' data.
5. WHEN a payment lacks Settlement_Metadata (legacy row), THE settlement grouping SHALL place it in the "Unassigned" bucket rather than failing or attributing it to the wrong school.
6. THE central-collection tagging SHALL be applied on every initiation path; a regression test SHALL assert the Beanola collector marker is present for each path.
7. THE payment metadata SHALL NOT persist raw phone numbers; only `phone_hash` and `phone_last4` are permitted, consistent with existing payment-hardening rules.

### Requirement 8: Application Uniqueness by Canonical Program + Intake

**User Story:** As an operator, I want duplicate-application rules to key on canonical program and intake, so that a student cannot create overlapping active applications even when multiple schools offer the same canonical program.

#### Acceptance Criteria

1. WHEN Canonical_IDs are present, THE DuplicateChecker SHALL evaluate uniqueness on `(student identity, canonical program, intake)` rather than on school offering or legacy strings.
2. WHILE multiple schools offer the same Canonical_Program for the same Intake, THE system SHALL still treat them as one canonical slot for duplicate purposes for a given student identity.
3. WHERE the same account legitimately applies on behalf of a different person (different NRC/passport identity), THE DuplicateChecker SHALL permit the additional application where the existing identity rules allow.
4. WHEN an existing application for the same canonical program + intake is in a terminal status (`rejected`, `withdrawn`, `expired`, `enrolled`, `enrollment_expired`), THE DuplicateChecker SHALL NOT block a new application.
5. WHEN Canonical_IDs are absent (legacy rows), THE DuplicateChecker SHALL preserve the previous string-keyword filter shape for backward compatibility.
6. THE duplicate check SHALL run both at draft creation and at submission.

### Requirement 9: Database Migration Readiness and Backfill Validation

**User Story:** As a platform engineer, I want the multi-tenant schema migration applied and validated against real Postgres, so that canonical IDs, tenant tables, and uniqueness constraints exist before tenant logic is trusted in production.

#### Acceptance Criteria

1. THE Tenant_Migration SHALL remain additive and idempotent: new tables, nullable columns, and indexes only, with best-effort backfills, and SHALL NOT drop or rewrite existing columns.
2. WHEN preparing to apply the Tenant_Migration, THE process SHALL first confirm the Migration_History_Prerequisite is applied, take a database backup, and run a dry run before applying to a staging copy.
3. WHEN the Tenant_Migration is applied to staging, THE process SHALL run the documented post-migration validation SQL (canonical program counts, offerings without canonical link, applications missing each Canonical_ID, duplicate hostnames, duplicate slugs, duplicate active memberships) and SHALL produce a manual exception report for legacy applications that cannot be linked.
4. THE not-valid foreign keys introduced by the Tenant_Migration SHALL be validated only after backfill issues are resolved, not before.
5. WHEN the migration and backfill complete, THE system SHALL verify that legacy applications with null Canonical_IDs remain readable, that new application creation writes all four Canonical_IDs, and that Official_Documents generated before the migration remain readable and unchanged.
6. THE database SHALL enforce at most one active membership per `(user_id, institution_id)` via the partial unique index, and supporting indexes for tenant application IDs, membership lookup, grant lookup, and template lookup SHALL be present.
7. THE migration SHALL be accompanied by a documented rollback note clarifying that additive tables/columns are safe to leave in place on rollback.

### Requirement 10: Student Program-First Wizard UX

**User Story:** As a student, I want to choose a program and intake first, then clearly confirm my assigned school and fee before paying, so that I always know which school I am applying to and what I owe.

#### Acceptance Criteria

1. THE student wizard SHALL order steps as: (1) program and intake, (2) assigned school review, (3) personal details, (4) education and documents, (5) payment, (6) review and submit.
2. WHEN a student selects a Canonical_Program and Intake, THE wizard SHALL send `program_id` and `intake_id` to the backend and SHALL display the assigned school, fee, required documents, and school contact returned by the backend before payment.
3. THE wizard SHALL NOT make the payment step reachable until assignment and fee are resolved.
4. WHEN assignment fails, THE wizard SHALL present a recoverable path (choose another intake, contact admissions, or join an interest list) and SHALL NOT dead-end the student.
5. WHILE a payment-bearing or settings form can lose work, THE wizard SHALL preserve auto-save and protect dirty state on navigation and `beforeunload`, consistent with existing admissions guardrails.
6. WHEN the request is on a White_Label_Portal, THE wizard brand and copy SHALL derive from runtime context (not hard-coded school names) and the offering choices SHALL be filtered to that institution.
7. THE wizard SHALL use canonical UI primitives (`PageShell`, `SectionCard`, `ErrorDisplay`, `EmptyState`, `Button asChild`), Lucide icons, WCAG AA contrast, and ≥44×44px touch targets, and SHALL NOT introduce purple gradients, gradient text, glassmorphism on product surfaces, nested cards, or emoji icons.

### Requirement 11: Admin Onboarding and Dashboard UX

**User Story:** As an operator, I want a clear tenant onboarding workflow and scope-correct dashboards, so that I can configure schools confidently and never see misleading cross-school totals.

#### Acceptance Criteria

1. THE admin tenant onboarding UI SHALL present an information architecture covering institutions, domains, branding assets, offerings, assignment rules, required documents, document templates, staff access, grants, settlement settings, and audit.
2. THE tenant onboarding UI SHALL support create, list, edit, and deactivate for tenant child resources (domains, assets, templates, required documents, memberships, grants), surfacing slug/code/domain collisions and asset upload success/failure states.
3. THE tenant onboarding UI SHALL provide a "Test routing" simulator whose result matches the real Assignment_Service exactly for the same inputs.
4. WHEN a Super_Admin views the dashboard, THE dashboard SHALL show cross-school widgets (applications by status/institution, payments by institution/offering/currency, pending documents by institution, capacity pressure, routing errors, recent tenant config changes).
5. WHEN School_Staff view the dashboard, THE dashboard SHALL show only scoped widgets (applications assigned to their school, awaiting review, missing documents, payment status, upcoming interviews, capacity/waitlist for their offerings).
6. WHEN a user has no school scope, THE dashboard SHALL render an explicit "No school access assigned" state with a support path and SHALL NOT show global zero counts.
7. THE tenant onboarding UI SHALL replace raw JSON rule entry with structured rule builders for countries, exclusions, priorities, and capacities, and SHALL provide template version diff/clone and an Official_Document preview where feasible.

### Requirement 12: Backward Compatibility and Platform Guardrails

**User Story:** As a platform owner, I want the multi-tenant conversion to preserve every existing contract and guardrail, so that production data, auth, payments, and admissions flows keep working throughout the rollout.

#### Acceptance Criteria

1. THE conversion SHALL preserve all existing `/api/v1/...` routes, filenames, and the `{"success": true, "data": ...}` response envelope, and authenticated list endpoints SHALL return the envelope (not raw lists).
2. THE conversion SHALL keep cookie-based auth, CSRF protection on state-changing requests, refresh-token rotation, and JTI blacklisting intact.
3. THE conversion SHALL preserve admissions auto-save, draft survival across refresh/reconnect, mobile-first usability, and the Lenco mobile-money-first payment UX, and SHALL NOT reintroduce retired pre-Lenco payment UX.
4. THE conversion SHALL NOT log PII, secrets, resume/document contents, or raw phone numbers, and SHALL keep audit trails free of plaintext PII.
5. THE conversion SHALL keep RBAC role hierarchy intact (`super_admin > admin > reviewer > student`) and SHALL NOT let admins assign roles above their own or infer cross-school access from role alone.
6. WHERE existing drift-guard and canonical-truth tests exist, THE conversion SHALL keep them green and SHALL add new drift guards for any new frontend-mirrored tenant concept.
7. THE conversion SHALL run the repo's verification gates for changed areas: backend `pytest` + `manage.py check` + schema generation when backend changes; admissions `bun run type-check`, `bun run test`, `bun run lint`, and build when frontend changes.

### Requirement 13: Tenant Observability and Audit

**User Story:** As an operator, I want auditable records and observable signals for tenant-sensitive actions, so that cross-tenant access, routing failures, and configuration changes can be detected and investigated.

#### Acceptance Criteria

1. WHEN assignment decisions, tenant configuration changes (institution/domain/asset/template/required-document/membership/grant create/update/deactivate), asset uploads, Official_Document generation, or access-scope denials occur, THE system SHALL write an Audit_Event including actor, action, target institution/application id, and relevant non-PII metadata.
2. THE system SHALL expose a Super_Admin-only view of recent tenant configuration changes and routing-failure events for operational review.
3. WHEN a routing failure (`NO_ELIGIBLE_OFFERING`) occurs, THE system SHALL record it with the canonical program, intake, and residency inputs so operators can fix coverage gaps.
4. THE Audit_Event payload SHALL NOT include full phone numbers, full NRC/passport numbers, or document contents; only hashed or masked identifiers are permitted.
5. WHERE a per-institution audit view is exposed to School_Staff, THE view SHALL be scoped to the caller's institutions only.

### Requirement 14: Correctness Property Coverage

**User Story:** As a quality engineer, I want property-based and scoped-access tests proving the canonical-ID and isolation invariants, so that cross-tenant leakage and string-based regressions are caught automatically.

#### Acceptance Criteria

1. THE test suite SHALL include `Assignment_Service` tests covering priority ordering, deterministic ties, residency/country/nationality allow and block, archived-offering read vs new-application deny, and capacity exhaustion.
2. THE test suite SHALL include `Access_Scope_Service` tests using real DB fixtures covering membership scope, grant scope (institution/offering/application), grant expiry, and queryset filtering for applications, payments, and documents.
3. THE test suite SHALL include cross-tenant isolation tests proving School_Staff cannot read, count, export, download, verify, receipt, or configure another school's data, and that out-of-scope lookups are indistinguishable from not-found.
4. THE test suite SHALL include `Institution_Context_Service` tests for uppercase host, port suffixes, inactive domains, inactive institutions, and hostname collisions.
5. THE test suite SHALL include Official_Document tests for template/asset version snapshotting, token allowlist + injection escaping, missing-template fallback, and asset MIME/magic-byte validation per allowed type.
6. THE test suite SHALL include payment Settlement_Metadata tests for every initiation path and settlement grouping tests for scoped school admins including the "Unassigned" bucket.
7. THE test suite SHALL include migration idempotency and backfill tests, and SHALL verify legacy rows with null Canonical_IDs remain readable while new applications write all four IDs.
8. THE frontend test suite SHALL include program-first wizard, white-label host filtering/branding, assigned-school checkpoint before payment, tenant onboarding create/update/deactivate, asset upload success/failure, and no-scope empty-state tests.
