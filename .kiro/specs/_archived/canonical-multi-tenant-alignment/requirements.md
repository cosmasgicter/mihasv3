# Requirements Document

## Introduction

The platform has moved from a MIHAS/KATC admissions application toward a
Beanola-owned multi-tenant admissions platform, but the implementation is not
yet fully aligned. The codebase contains real multi-tenant foundations alongside
legacy admission assumptions, duplicate admin route definitions, frontend PDF
generation paths, compatibility re-export layers, split draft/session logic, and
tenant configuration surfaces that need a single enterprise contract.

This spec is the canonical alignment plan for finishing the transition. It does
not replace the historical work in these specs; it consolidates the remaining
work into one implementation authority:

- `.kiro/specs/multi-tenant-beanola-admissions/`
- `.kiro/specs/multi-tenant-beanola-remediation/`
- `.kiro/specs/enterprise-tenant-authority/`
- `.kiro/specs/beanola-production-readiness/`
- `.kiro/specs/single-source-of-truth-consolidation/`
- `.kiro/specs/duplicate-deprecated-consolidation/`

The implementation must preserve completed canonical work and remove or
delegate stale paths. The goal is a clean, mobile-first, fully functional,
future-ready enterprise admissions product where Beanola Technologies owns the
platform and every school is tenant data.

## Glossary

- **Beanola**: Platform owner and neutral shared-portal brand.
- **Tenant / Institution / School**: A university, college, or training school
  onboarded onto the platform.
- **Super_Admin**: Beanola platform operator with global authority.
- **Tenant_Admin**: Admin scoped to one or more institutions through membership
  or explicit grants.
- **Approver / Reviewer**: Staff user scoped to assigned applications,
  offerings, or institutions.
- **Student**: Applicant who owns applications, drafts, uploads, payments, and
  student-visible official documents.
- **Canonical_Program**: Program concept a student chooses.
- **Offering**: School-specific program availability, fee, intake, and rule set.
- **Official_Document**: Backend-generated and stored tenant-branded PDF with
  provenance, fingerprint, and authorization.
- **Document_Profile**: Tenant-owned document content and layout configuration.
- **Institution_Asset**: Tenant logo, signature, seal, or other brand artifact.
- **Route_Registry**: Single frontend source of truth for product routes.
- **Compatibility_Layer**: Re-export, alias, or old endpoint retained only to
  keep older callers working while canonical imports/routes are migrated.

## Hard Constraints

1. Beanola is the platform owner. MIHAS and KATC are tenant records only.
2. Tenant data boundaries are enforced by the backend, not by hidden frontend
   controls.
3. New code must resolve authority through `AdminCapabilityService` and scope
   through `AccessScopeService` or documented canonical helpers.
4. Product admin UI uses `/admin/*`; Django operational admin uses
   `/beanola-admin-panel/`; API admin uses `/api/v1/admin/*`.
5. Students apply program-first. Institution assignment, required documents,
   fees, document templates, tenant branding, and settlement metadata are
   backend decisions.
6. Official student/admin documents are generated by the backend official
   document pipeline only.
7. Frontend PDF renderers may exist only as dev previews or non-official
   previews and must not be reachable from official user flows.
8. No new arbitrary DOCX/PDF merge engine is allowed.
9. Legacy columns and historical data remain readable.
10. Out-of-scope tenant reads must not leak object existence.
11. Every visible admin action must map to a backend capability.
12. Every major workflow must be usable on mobile first.

## Requirements

### Requirement 1: Canonical Business Contract

**User Story:** As Beanola, I want one documented business contract for the
platform, so that future implementation work cannot drift back into MIHAS/KATC
single-tenant assumptions.

#### Acceptance Criteria

1. The system SHALL document Beanola as platform owner and all schools as
   tenant data.
2. The system SHALL document student program-first application flow as the
   canonical flow.
3. The system SHALL document Super_Admin, Tenant_Admin, Approver/Reviewer, and
   Student authority.
4. The system SHALL document official document authority as backend-only.
5. The system SHALL document the distinction between product admin UI,
   operational Django admin, and admin API routes.
6. The system SHALL document the deprecated or compatibility-only paths that
   must not be used for new work.
7. The system SHALL update steering docs when implementation changes the
   canonical contract.

### Requirement 2: Canonical Route Registry

**User Story:** As a frontend engineer, I want one route registry, so that
admin links, redirects, mobile nav, desktop nav, route guards, SEO metadata, and
auth redirects cannot contradict each other.

#### Acceptance Criteria

1. The frontend SHALL expose one product route registry for all application
   routes.
2. The route registry SHALL include route id, path, guard, required capability,
   skeleton type, navigation metadata, and redirect aliases.
3. `routes/config.tsx`, admin nav, mobile nav, desktop sidebar, auth redirect,
   not-found links, breadcrumbs, prefetching, and SEO path declarations SHALL
   consume the registry or a typed derivative of it.
4. The system SHALL distinguish `/admin/tenants` from `/admin/tenants/new`.
5. The system SHALL distinguish `/admin/*` product routes from
   `/beanola-admin-panel/` Django admin.
6. Tests SHALL fail when hard-coded `/admin/` route strings are added outside
   allowed route-registry files.
7. Route aliases SHALL be intentional and documented, not scattered.

### Requirement 3: Admin Information Architecture

**User Story:** As an admin user, I want to see only the surfaces relevant to my
authority, so that tenant admins are not confused by platform controls and
super admins can manage the full platform.

#### Acceptance Criteria

1. Super_Admin SHALL see the Beanola platform console, tenant list, tenant
   onboarding, domain management, cross-tenant reporting, and global settings.
2. Tenant_Admin SHALL see a "My School" console for assigned institutions only.
3. Tenant_Admin SHALL NOT see tenant creation controls.
4. Tenant_Admin SHALL NOT see data for other tenants unless an explicit grant
   exists.
5. Approver/Reviewer SHALL see review queues and documents only for assigned
   scope.
6. Admin empty states SHALL explain missing permissions without exposing other
   tenants.
7. Mobile and desktop admin navigation SHALL show the same authorized
   destinations.

### Requirement 4: Backend Authorization Enforcement

**User Story:** As a platform owner, I want every tenant-sensitive backend path
to enforce scope centrally, so that frontend mistakes cannot become security
incidents.

#### Acceptance Criteria

1. Tenant-sensitive endpoint code SHALL NOT grant authority based only on raw
   role string checks.
2. Tenant-sensitive object lookup SHALL filter scope before retrieving the
   object where feasible.
3. Super_Admin SHALL have global access only through the documented capability
   path.
4. Tenant_Admin and Approver/Reviewer access SHALL be derived from memberships
   and grants.
5. Out-of-scope reads SHALL return the canonical non-leaking error shape.
6. Out-of-scope mutations SHALL be denied before serializer save.
7. Tests SHALL cover direct API calls for tenant creation, tenant updates,
   document profile changes, application review, payment verification, and
   exports.

### Requirement 5: Tenant Onboarding Lifecycle

**User Story:** As a Super_Admin, I want to onboard a school end to end, so that
the school can go live without code changes.

#### Acceptance Criteria

1. Super_Admin SHALL create a tenant institution from the UI.
2. Tenant creation SHALL create or require a launch-readiness checklist.
3. Tenant onboarding SHALL support logo, signature, and seal upload.
4. Tenant onboarding SHALL support document profiles by document type.
5. Tenant onboarding SHALL support required documents.
6. Tenant onboarding SHALL support program/offering and intake assignment.
7. Tenant onboarding SHALL support fees and settlement metadata where present.
8. Tenant onboarding SHALL support domain configuration and simulation.
9. Tenant onboarding SHALL support tenant admin invitation.
10. A tenant SHALL NOT be marked launch-ready until required configuration is
    complete.
11. Tenant readiness SHALL be testable through a backend endpoint or management
    command.

### Requirement 6: Tenant Domain Resolution

**User Story:** As Beanola, I want domains to resolve deterministically to
tenant context, so that shared and white-label portals work safely.

#### Acceptance Criteria

1. Unknown domains SHALL resolve to neutral Beanola context.
2. Disabled, unverified, or conflicting domains SHALL resolve fail-closed.
3. Active verified domains SHALL resolve to exactly one tenant.
4. Domain simulation SHALL be available before activation.
5. Domain resolution SHALL influence catalog context, branding, and offering
   filtering.
6. Domain changes SHALL be audited.
7. Domain setup steps SHALL be documented in a production runbook.

### Requirement 7: Program-First Application Assignment

**User Story:** As a student, I want to apply for a program without worrying
about internal tenant routing, so that the system handles institution assignment
automatically.

#### Acceptance Criteria

1. Student-facing program selection SHALL prioritize canonical program
   language over institution-specific implementation details.
2. The backend SHALL assign an offering/institution deterministically using the
   canonical assignment service.
3. Assignment failure SHALL return recoverable guidance, not a generic error.
4. Assigned offering SHALL determine fees, required documents, document
   profiles, and application numbering.
5. Multiple tenants offering the same canonical program SHALL be handled by
   explicit rules such as domain, priority, intake availability, capacity, and
   eligibility.
6. Assignment decision metadata SHALL be available to admins for support and
   audit.

### Requirement 8: Application Start, Resume, And Draft Contract

**User Story:** As a student, I want to choose between starting a new
application and resuming an existing one, so that the wizard never silently
loads the wrong application.

#### Acceptance Criteria

1. The frontend SHALL support explicit `new`, `resume draft`, and `view
   application` modes.
2. `Start New Application` SHALL clear local wizard state for that session and
   SHALL NOT auto-load any existing draft.
3. `Continue Draft` SHALL load only the selected draft or selected application.
4. LocalStorage, sessionStorage, server draft rows, and application rows SHALL
   follow one documented lifecycle.
5. The backend SHALL either support true multi-draft APIs or the UI SHALL
   clearly present a single active draft model; ambiguous hybrid behavior is
   not allowed.
6. Duplicate detection SHALL not silently adopt an old application without an
   explicit user choice.
7. Deleting a draft SHALL clear local cache, session flags, dashboard state, and
   backend draft state consistently.
8. Tests SHALL cover start-new with existing local draft, start-new with
   existing server draft, resume selected draft, duplicate conflict, delete
   draft, and payment-linked draft protection.

### Requirement 9: Official Document Authority

**User Story:** As a student or admin, I want official documents to be generated
from tenant configuration, so that every document is branded and legally
consistent for the assigned school.

#### Acceptance Criteria

1. Official document generation SHALL use backend tenant profiles and assets.
2. Official document generation SHALL use resolved institution, program,
   intake, fees, required documents, signatory, logo, signature, and seal.
3. Official documents SHALL store provenance and input fingerprint.
4. Repeated generation SHALL reuse the current official version when inputs have
   not changed.
5. Missing document profile or missing required asset SHALL produce a clear
   setup error.
6. No production official document flow SHALL call frontend `@/lib/pdf`
   generators.
7. Legacy official document endpoints SHALL delegate to the official document
   service or become deprecated compatibility endpoints.
8. Admin and student document actions SHALL expose queued, generating, ready,
   failed, and setup-required states.

### Requirement 10: Tenant Document Configuration

**User Story:** As a Super_Admin or authorized tenant admin, I want tenant
document templates, content, signatures, and required documents to be configured
without code changes.

#### Acceptance Criteria

1. The admin UI SHALL allow editing document profiles per document type.
2. The admin UI SHALL allow scoped profiles by institution, program, offering,
   intake, or default policy where supported.
3. The admin UI SHALL validate allowed tokens and structured content.
4. The admin UI SHALL show active asset versions used by documents.
5. SVG asset limitations SHALL be surfaced before upload or during validation.
6. Profile versioning SHALL preserve old generated documents.
7. Document preview SHALL use backend sample rendering or clearly dev-only
   frontend preview.
8. Tenant document configuration SHALL be included in launch readiness.

### Requirement 11: Legacy Document And PDF Decluttering

**User Story:** As a developer, I want one official document path, so that old
PDF utilities cannot produce inconsistent records.

#### Acceptance Criteria

1. `useDocumentGeneration.ts` SHALL be removed, renamed to a dev-only preview
   hook, or rewritten to call official backend services.
2. `usePaymentReceipt.ts` SHALL use official backend receipt/document behavior
   or be explicitly non-official.
3. `applicationService.generateApplicationSlip` SHALL delegate to official
   documents or be deprecated with no production callers.
4. Frontend PDF imports SHALL be limited to dev-preview routes or documented
   non-official previews.
5. The build SHALL prove no student/admin official document path imports
   `@/lib/pdf`.
6. Legacy document endpoints SHALL be tested for delegation or deprecation.

### Requirement 12: Tenant Branding And Platform Branding

**User Story:** As Beanola, I want platform pages and tenant pages to use the
correct brand, so that the platform no longer feels like MIHAS/KATC software.

#### Acceptance Criteria

1. Shared platform surfaces SHALL use Beanola branding.
2. Tenant surfaces SHALL use the resolved tenant brand when tenant context is
   available.
3. MIHAS/KATC strings SHALL be allowed only in seed data, tests, tenant
   fixtures, historical records, or explicit legacy allowlists.
4. Application numbers and payment references SHALL not default to MIHAS or
   KATC in generic flows.
5. Emails and notifications SHALL resolve tenant brand first, then Beanola.
6. Brand drift tests SHALL fail on unapproved MIHAS/KATC generic references.

### Requirement 13: De-AI Duplicate And Compatibility Cleanup

**User Story:** As a maintainer, I want duplicate and compatibility-only code
removed or clearly bounded, so that the codebase is understandable and future
agents stop layering work on top of stale paths.

#### Acceptance Criteria

1. Every compatibility re-export SHALL have a tracked removal decision or be
   removed.
2. Tests SHALL import canonical modules where practical.
3. Duplicate helpers, hooks, services, route maps, and document flows SHALL be
   consolidated.
4. Comments SHALL describe current business rules rather than historical agent
   implementation details.
5. Deprecated modules SHALL fail lint/import checks if used by new production
   code.
6. A duplicate-code audit SHALL be added to the runbook or CI gate.

### Requirement 14: Frontend Mobile-First Product Polish

**User Story:** As a student or admin on a phone, I want every important
workflow to be usable and polished, so that the platform feels production-grade.

#### Acceptance Criteria

1. Student application wizard SHALL be mobile-first with clear progress,
   recoverable errors, autosave status, and explicit new/resume mode.
2. Student dashboard SHALL clearly separate new application, continue draft,
   submitted applications, payments, documents, interviews, and communications.
3. Admin dashboards and tables SHALL avoid horizontal scroll traps on mobile.
4. Tenant onboarding SHALL work on mobile with step validation and save/resume.
5. Touch targets SHALL be at least 44px.
6. Icon-only controls SHALL have accessible labels.
7. Text SHALL not overflow buttons, cards, tables, nav, or dialogs.
8. Loading and empty states SHALL be specific and actionable.
9. The UI SHALL use consistent tokens, spacing, density, icons, and component
   primitives.
10. The UI SHALL avoid decorative gradients, nested cards, and marketing-style
    composition inside operational admin tools.

### Requirement 15: Backend/API Contract Sync

**User Story:** As an implementation team, I want frontend services and backend
serializers to share a stable contract, so that UI features do not depend on
guessed shapes or stale endpoints.

#### Acceptance Criteria

1. Tenant service methods SHALL map to backend admin endpoints one-to-one.
2. API response envelopes SHALL be consistent for list, detail, create, update,
   delete, validation, authorization, and setup-required states.
3. Frontend types SHALL match backend serializer fields.
4. OpenAPI snapshots or contract tests SHALL cover tenant onboarding, document
   profiles, assets, domains, offerings, drafts, official documents, scope, and
   capabilities.
5. Legacy endpoint callers SHALL be identified and removed or delegated.

### Requirement 16: Performance And Operational Readiness

**User Story:** As a platform operator, I want the platform to run reliably on
the current small EC2 deployment while being ready to scale, so that production
does not fail under normal usage.

#### Acceptance Criteria

1. Admin dashboard queries SHALL be cached or consolidated where safe.
2. Application list payment summary SHALL avoid excessive correlated
   subqueries.
3. Catalog data SHALL use server-side cache with tenant-scope-safe invalidation.
4. Hot list endpoints SHALL avoid unnecessary counts where pagination can be
   cursor-based.
5. Heavy Celery tasks SHALL not starve document/payment/SLA work on a single
   worker.
6. Static assets SHALL be compressed and cached appropriately.
7. Deploy scripts SHALL prevent recurring disk exhaustion.
8. Production runbooks SHALL include migration, backup, rollback, disk, worker,
   and tenant-readiness checks.

### Requirement 17: Observability, Audit, And Supportability

**User Story:** As Beanola support, I want tenant operations and assignment
decisions to be traceable, so that support issues can be solved without unsafe
database digging.

#### Acceptance Criteria

1. Tenant create/update, domain changes, asset uploads, document profile
   changes, grants, membership changes, assignment decisions, official document
   generation, and denied access attempts SHALL be auditable.
2. Audit metadata SHALL avoid secrets, raw document bodies, and unnecessary PII.
3. Admin support views SHALL expose assignment decision summaries where
   appropriate.
4. Official document failures SHALL be visible to authorized admins.
5. Tenant readiness failures SHALL be explainable and actionable.

### Requirement 18: Verification And Release Gates

**User Story:** As the owner, I want proof that the platform is aligned before
launch, so that "done" means the actual user flows work end to end.

#### Acceptance Criteria

1. The implementation SHALL include unit, integration, property, contract, and
   E2E coverage for the critical flows.
2. CI SHALL include drift guards for routes, brand, document flows, tenant
   scope, frontend PDF official usage, and migration placement.
3. E2E tests SHALL cover student start-new, resume draft, submit application,
   payment/document path, super-admin tenant onboarding, tenant-admin isolation,
   domain simulation, and official document generation.
4. Production smoke runbook SHALL verify `/`, `/student/dashboard`,
   `/admin/dashboard`, `/admin/tenants`, `/admin/tenants/new` as Super_Admin,
   `/beanola-admin-panel/`, API health, migrations, Celery, Redis, Postgres,
   and generated PDFs.
5. The release SHALL not be considered production-ready until all non-negotiable
   acceptance criteria pass.
