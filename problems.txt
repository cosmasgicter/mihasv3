1. `confirmed-bug` Payment form sends the wrong phone contract on card/widget payments
Severity: High

Evidence:
- [apps/admissions/src/components/student/PaymentForm.tsx](/home/cosmas/Downloads/mihasv3/apps/admissions/src/components/student/PaymentForm.tsx:100) returns customer details as `{ fullName, email, phone: initialPhone }`.
- [apps/admissions/src/components/student/PaymentForm.tsx](/home/cosmas/Downloads/mihasv3/apps/admissions/src/components/student/PaymentForm.tsx:167) normalizes mobile-money input to a strict 10-digit local number before posting `/payments/mobile-money/`.
- [apps/admissions/src/hooks/useApplicationPaymentAction.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/hooks/useApplicationPaymentAction.ts:135) posts `customer_phone: customer.phone || ''` to `/payments/initiate/`.
- [apps/admissions/src/hooks/useApplicationPaymentAction.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/hooks/useApplicationPaymentAction.ts:166) also forwards the same raw value into `customerPhone` for the Lenco widget.
- [apps/admissions/src/pages/student/applicationWizard/steps/PaymentStep.tsx](/home/cosmas/Downloads/mihasv3/apps/admissions/src/pages/student/applicationWizard/steps/PaymentStep.tsx:133) passes `watch('phone')` into `PaymentForm`.
- [apps/admissions/src/pages/student/Payment.tsx](/home/cosmas/Downloads/mihasv3/apps/admissions/src/pages/student/Payment.tsx:210) passes `app.phone` into `PaymentForm`.

Impact:
- Mobile money and card flows do not use the same phone contract.
- A stored applicant phone such as `260969992299` or `+260...` can leak into card/widget initiation even though the business rule is a 10-digit local applicant phone.
- This matches the production symptom already observed by the user.

Recommended action:
- Normalize customer phone once in `PaymentForm` or `useApplicationPaymentAction` and use the normalized local 10-digit value for every payment channel.
- Add a focused test proving that both mobile-money and card/widget initiation receive the same normalized phone value.

2. `confirmed-bug` Frontend still sends stale `customer_*` fields that the backend `/payments/initiate/` contract ignores
Severity: Medium

Evidence:
- [apps/admissions/src/hooks/useApplicationPaymentAction.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/hooks/useApplicationPaymentAction.ts:137) sends `customer_email`, `customer_name`, and `customer_phone`.
- [backend/apps/documents/views.py](/home/cosmas/Downloads/mihasv3/backend/apps/documents/views.py:503) documents only `application_id` in the request schema.
- [backend/apps/documents/views.py](/home/cosmas/Downloads/mihasv3/backend/apps/documents/views.py:513) reads only `application_id = request.data.get("application_id")`.
- [backend/apps/documents/views.py](/home/cosmas/Downloads/mihasv3/backend/apps/documents/views.py:545) initiates the payment with only `application_id` and `user_id`.

Impact:
- The frontend and backend are operating on different contracts.
- The frontend believes customer metadata is part of the initiation path, but the backend ignores it completely.
- This is a classic stale-client symptom after backend refactors.

Recommended action:
- Either remove the stale frontend fields, or formally extend the backend serializer/service contract and actually use them.
- Add a contract test that fails if frontend initiation fields diverge from backend schema again.

3. `confirmed-bug` Student receipt download bypasses canonical backend receipt APIs and fabricates receipt data in the browser
Severity: High

Evidence:
- [apps/admissions/src/components/student/DownloadReceiptButton.tsx](/home/cosmas/Downloads/mihasv3/apps/admissions/src/components/student/DownloadReceiptButton.tsx:21) calls `generateReceipt(applicationId)` in the browser.
- [apps/admissions/src/hooks/usePaymentReceipt.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/hooks/usePaymentReceipt.ts:54) fetches payments from `/payments/?application_id=...`.
- [apps/admissions/src/hooks/usePaymentReceipt.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/hooks/usePaymentReceipt.ts:68) fabricates `receiptNumber` with `generateReceiptNumber()` when absent.
- [apps/admissions/src/hooks/usePaymentReceipt.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/hooks/usePaymentReceipt.ts:107) generates the PDF entirely client-side.
- [backend/apps/documents/views.py](/home/cosmas/Downloads/mihasv3/backend/apps/documents/views.py:370) already exposes canonical `GET /api/v1/payments/{id}/receipt/`.
- [backend/apps/documents/urls.py](/home/cosmas/Downloads/mihasv3/backend/apps/documents/urls.py:114) wires the payment receipt endpoint.
- [backend/apps/applications/document_views.py](/home/cosmas/Downloads/mihasv3/backend/apps/applications/document_views.py:211) also exposes `POST /api/v1/applications/{id}/finance-receipt/` for finance workflows.

Impact:
- The student UI is not using the canonical backend receipt contract.
- Receipt numbers and receipt contents can drift from backend truth.
- Auditability is weakened because a client can synthesize receipt PDFs from partial or stale data.

Recommended action:
- Stop generating official receipts client-side.
- Use the backend receipt endpoint as the only source of truth, then render or download the server-returned receipt artifact/data.
- Remove synthetic `generateReceiptNumber()` from the official receipt flow.

4. `zero-day-class-risk` File-upload auth recovery path still treats all `403` responses as session expiry
Severity: High

Evidence:
- [apps/admissions/src/pages/student/applicationWizard/hooks/useApplicationFileUploads.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/pages/student/applicationWizard/hooks/useApplicationFileUploads.ts:59) defines `isAuthError` as `401 || 403`.
- [apps/admissions/src/pages/student/applicationWizard/hooks/useApplicationFileUploads.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/pages/student/applicationWizard/hooks/useApplicationFileUploads.ts:71) retries `/auth/session/` and converts repeated `401/403` into "session expired".
- [apps/admissions/src/lib/sessionHardening.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/lib/sessionHardening.ts:21) explicitly distinguishes generic permission-denial `403` from auth failure.

Impact:
- One stale path can still log out or interrupt an authenticated user on permission-denial style failures.
- This directly contradicts the central session-hardening rules and can create hard-to-reproduce upload failures.

Recommended action:
- Make upload/session recovery reuse the same 403 classification logic as the main API client.
- Add a regression test: generic `403 INSUFFICIENT_PERMISSIONS` must not trigger auth-expired UX.

5. `zero-day-class-risk` `secureStorage` encryption is not initialized anywhere, so the "secure storage" guarantee is false
Severity: High

Evidence:
- [apps/admissions/src/lib/secureStorage.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/lib/secureStorage.ts:157) requires `init(sessionToken)` to derive the encryption key.
- Repository search shows no runtime call to `secureStorage.init(...)`.
- `rg -n "secureStorage\\.init\\(|clearSession\\(|secureStorage" apps/admissions/src -S` only found the definition plus cleanup usages.
- [apps/admissions/src/contexts/AuthContext.tsx](/home/cosmas/Downloads/mihasv3/apps/admissions/src/contexts/AuthContext.tsx:75) clears secure storage on logout.
- [apps/admissions/src/hooks/auth/useSessionListener.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/hooks/auth/useSessionListener.ts:214) also clears it on session events.

Impact:
- The project contains a security-sensitive utility that looks active but never actually enables encryption.
- If developers assume local drafts or session-adjacent data are protected by this layer, that assumption is wrong.
- This is "zero-day-class" because it is silent and gives a false security guarantee.

Recommended action:
- Either initialize secure storage immediately after successful authentication, or remove the abstraction and document the real storage guarantees.
- Add a test proving `secureStorage.initialized === true` after login and that sensitive payloads are encrypted at rest if this feature is kept.

7. `suspicious-stale-path` Receipt hook depends on a payments list endpoint shape that is already known to be migration-sensitive
Severity: Medium

Evidence:
- [apps/admissions/src/hooks/usePaymentReceipt.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/hooks/usePaymentReceipt.ts:56) fetches `/payments/?application_id=...`.
- [apps/admissions/src/hooks/usePaymentReceipt.ts](/home/cosmas/Downloads/mihasv3/apps/admissions/src/hooks/usePaymentReceipt.ts:59) accepts multiple response shapes and falls back broadly.
- The backend canonical receipt flow is keyed by `payment_id`, not "list then guess the right payment".

Impact:
- Broad response-shape tolerance usually indicates the frontend is compensating for unstable or previously changed contracts.
- This increases the chance of stale data selection, wrong receipt source, and future silent breakage.

Recommended action:
- Stop using "list then infer" for receipt generation.
- Drive receipts from a specific verified payment record and canonical backend route only.
