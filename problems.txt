You are an agentic engineering audit team working on the MIHAS admissions system repository. Your task is to analyze the admissions frontend, backend, and database paths with facts only. Do not assume behavior. Prove every claim from code, migrations, tests, configs, logs, or reproducible command output.

Primary objective:
Audit every relevant file in the admissions system and produce a factual, evidence-backed report of defects, race conditions, canonical-path violations, Redis/cache weaknesses, admin-to-student communication gaps, and payment-system risks. The payment system must be traced end to end.

Scope:
- Frontend admissions app: `apps/admissions`
- Backend admissions API: `backend`
- Database schema, migrations, and schema verification files: `backend/migrations`, `backend/apps/**/models.py`, migration scripts, and any DB-related docs or tests that describe actual runtime behavior
- Shared packages only when imported by admissions frontend/backend code
- Tests that prove or contradict the intended behavior
- Configuration files that affect auth, API routing, Redis/cache, payments, CORS, cookies, CSRF, security headers, background jobs, queues, and deployment

Current-change awareness:
- This repository may contain many uncommitted, newly added, modified, and deleted files. Treat the working tree as the source of truth for this audit.
- Before inspecting business logic, run `git status --short`, `git diff --name-status`, and `git diff --stat`.
- Include untracked files in the audit. Use `rg --files` and compare it with `git status --short` so new files are not missed.
- Deleted files matter. For each deleted file, verify whether any import, route, export, test, or runtime reference still points to it.
- Modified files matter. For each modified file, inspect both the current file and the diff from `HEAD` when possible. Identify whether the new changes alter canonical paths, auth/session behavior, Redis/cache behavior, communication behavior, or payment behavior.
- New backend services matter. If files such as amendment, condition, enrollment, interview, waitlist, withdrawal, communication, templates, fee waiver, expiry, late application, reviewer assignment, or batch-operation services exist, treat them as first-class admissions flow code and trace their API, DB, task, notification, permission, and payment implications.
- New tests matter. If new tests exist, determine whether they prove the new behavior, only assert surface behavior, or encode assumptions that are not supported by implementation.
- Separate findings into:
  - `introduced-or-changed`: caused or exposed by current working-tree changes
  - `pre-existing`: present before the current changes
  - `unknown-origin`: cannot be attributed without more history
- Do not revert, clean, or rewrite user changes. Analysis only.

Non-negotiable rules:
- Do not make assumptions.
- Do not rely on memory or generic framework behavior unless you verify it in this repository or official framework documentation.
- If a fact is inferred, label it explicitly as an inference and cite the evidence that supports it.
- Every finding must include exact file paths and line numbers.
- Every finding must include a reproducible verification step or a precise reason why it cannot currently be reproduced.
- Do not claim a defect unless you can show the code path, data path, or test gap that makes it possible.
- Do not skip files because they look unrelated. If a file is in scope, classify it as relevant, indirectly relevant, or not relevant, and explain the classification briefly in the file inventory.
- Do not modify code unless the user explicitly asks for implementation after the audit. This task is analysis only.
- Work in parallel through multiple subagents. Each subagent must own a disjoint slice of the repository and return evidence, not opinions.

Required parallel subagents:
0. Change Impact and Deletion Agent
   - Owns the working-tree inventory, `git status --short`, `git diff --name-status`, `git diff --stat`, untracked files, deleted files, changed exports/imports, package/config changes, and route/build fallout.
   - Classify every changed path as frontend, backend, database/migration, test, config, docs/spec, or unknown.
   - For each deleted file, search for remaining references by import path, filename, exported symbol, route name, and test reference.
   - For each new file, identify its public API, callers, tests, and whether it is wired into runtime routes/tasks/services.
   - Produce a change-risk summary before other subagents finalize their reports.

1. Frontend Canonical Flow Agent
   - Owns `apps/admissions/src/pages`, `apps/admissions/src/components`, `apps/admissions/src/hooks`, `apps/admissions/src/contexts`, `apps/admissions/src/routes`, and frontend tests.
   - Trace student application creation, draft saving/deletion, wizard step movement, dashboard routing, payment routing, status views, admin views, and student communication UI.
   - Identify any path that bypasses the canonical application flow.
   - Check for React state races, stale query cache use, duplicate requests, refocus/session races, unmounted updates, stale localStorage/sessionStorage, and conflicting event listeners.

2. Backend Canonical Flow Agent
   - Owns `backend/apps`, `backend/config`, backend tests, serializers, views, services, tasks, middleware, permissions, and URL routing.
   - Trace application lifecycle from draft to submission to review to approval/rejection.
   - Include any new amendment, condition, enrollment, interview scheduling, waitlist, withdrawal, expiry, late application, reviewer assignment, batch operation, fee waiver, communication, and template services in the canonical lifecycle map if present.
   - Verify permissions and object ownership for student/admin/reviewer endpoints.
   - Identify race conditions in submissions, payment verification, status updates, document uploads, notifications, and background tasks.
   - Confirm that admin actions and student actions use one canonical service path where required.

3. Database and Migration Agent
   - Owns migrations, models, schema verification scripts, DB docs, and tests that validate schema behavior.
   - Map all tables used by admissions flows: users/profiles, applications, application drafts, documents, grades, payments, payment audits/status history, notifications, communications, intakes, programs, fees, sessions, tokens, and Redis-backed state if represented.
   - Verify constraints, indexes, foreign keys, uniqueness, nullability, status fields, amount fields, timestamps, and audit columns.
   - Identify schema drift between frontend types, backend serializers/models, and migrations.

4. Redis, Cache, Session, and Race Agent
   - Owns Redis/cache/session settings, query cache strategy, background jobs, rate limiting, polling, token refresh, CSRF handling, and any queue/task code.
   - Verify Redis usage is correct, namespaced, expiring where needed, non-lossy where required, and safe under concurrency.
   - Check cache invalidation for application changes, payment changes, admin review changes, communication updates, and auth refresh/logout.
   - Identify race conditions such as parallel token refreshes, stale dashboard data, duplicate payment initiation, double submission, duplicate draft creation, delayed webhook processing, retry storms, and cache poisoning.

5. Payment End-to-End Agent
   - Owns all Lenco/payment code across frontend, backend, DB, tests, configs, docs, and webhook/task handling.
   - Trace payment from fee resolution to frontend display, payment initiation, Lenco widget configuration, callback handling, backend payment record creation, verification, webhook idempotency, polling, status mapping, admin review, receipt/slip generation, and dashboard/status updates.
   - Verify mobile friendliness of payment surfaces using actual frontend code and responsive tests. If browser verification is possible, run desktop and mobile viewport checks and record evidence.
   - Verify amount integrity, currency handling, application ownership, idempotency keys/references, duplicate payment prevention, retry behavior, pending/failed/successful transitions, and webhook signature validation.
   - Confirm that old/unpaid/rejected fees use dedicated payment screens and do not force users through the wizard unless the application is truly a draft.

6. Admin-to-Student Communication Agent
   - Owns communication, notification, email, SMS, audit, admin action, and student-facing message paths.
   - Trace how admins communicate with potential students and existing applicants.
   - Verify delivery paths, permissions, templates, audit logs, retry behavior, error handling, read/unread state, notification preferences, and student visibility.
   - Identify gaps where an admin action changes application/payment state but the student is not clearly notified.

Coordination protocol:
- Start by building a repository map and a change map. Use fast file discovery such as `rg --files`, then inspect `git status --short`, `git diff --name-status`, and `git diff --stat`.
- The Change Impact and Deletion Agent must publish its change-risk summary first so other subagents can prioritize changed files and deleted-file fallout.
- Each subagent must produce:
  - files inspected
  - changed/new/deleted files inspected
  - canonical paths found
  - facts with citations
  - defects or risks with severity
  - tests or commands run
  - unanswered questions blocked by missing evidence
- Merge all subagent outputs into one final report.
- If two subagents disagree, resolve the conflict by citing code or tests. If unresolved, mark it as an open question with exact evidence on both sides.

- Required canonical flows:

  - education/grades/document upload

  - payment initiation
  - Lenco widget callback and verification
  - webhook handling and payment polling
  - receipt/slip generation
  - application submission
  - admin application review
  - admin payment review/status update
  - student dashboard/status refresh
  - admin-to-student messages/notifications
  - amendments and corrections, if present
  - conditional offers and condition clearance, if present

  - withdrawal, if present

Race-condition checklist:
- Double-click and duplicate submit handling
- Multiple tabs editing the same draft/application
- Local draft versus server draft conflicts
- Stale localStorage/sessionStorage references
- React Query cache invalidation races
- Polling versus mutation overwrite races
- Token refresh deduplication and auth-expired cascades
- CSRF refresh and retry races
- File upload and application ID creation ordering
- Payment initiation double calls
- Payment verification versus webhook update races
- Admin status update versus student submission/payment update races
- Background task retries and idempotency
- Redis key expiration and stale lock risks

Redis/cache requirements:
- Identify every Redis/cache use and every place Redis should be used but is not.
- Check key naming, tenant/user scoping, TTLs, lock ownership, atomicity, retry behavior, serialization, cleanup, and fallback behavior when Redis is unavailable.
- Verify cache invalidation after:
  - login/logout/session refresh
  - draft create/update/delete
  - application submit
  - document upload/verification
  - payment initiate/verify/webhook/poll
  - admin review/payment status update
  - notification/communication send
- Report stale-cache risks separately from confirmed stale-cache bugs.

Payment end-to-end requirements:
- Verify fee amount source of truth.
- Verify frontend never trusts user-supplied amount for final payment amount.
- Verify payment initiation checks application ownership and payable state.
- Verify Lenco references are unique and idempotent.
- Verify amount and currency are validated on verification/webhook.
- Verify webhook signature validation and idempotency.
- Verify pending payments are recoverable.
- Verify failed/rejected payments can be retried safely.
- Verify successful payments cannot be replayed to alter unrelated applications.
- Verify old unpaid fees and rejected payments use dedicated payment screens, not the wizard.
- Verify drafts do not appear as payable submitted applications.
- Verify mobile UI has usable tap targets, no overflow, readable status messages, and recovery when the widget script fails.
- Verify fee waivers, late fees, expired payments, payment expiry, settlement metadata, and manual/admin payment overrides if present in the current working tree.

Admin/student communication requirements:
- Verify every important admin action creates a student-visible outcome or documented notification path.
- Verify payment rejection includes a reason visible to the student.
- Verify interview scheduling/rescheduling/canceling notifies the student.
- Verify document rejection/approval feedback is visible.
- Verify message delivery failures are logged and retryable.
- Verify admin communication actions are audited.

Output format:

   - deleted files with remaining references
   - untracked files with wiring status
   - package/config/deployment changes
   - tests added or changed
   - highest-risk changes to inspect first

3. File inventory
   - grouped by frontend, backend, database/migrations, tests, config
   - mark each as relevant, indirectly relevant, or not relevant

4. Canonical flow map
   - one subsection per required flow
   - include exact file paths and line references

5. Findings
   - severity: Critical, High, Medium, Low
   - origin: introduced-or-changed, pre-existing, unknown-origin
   - category: canonical path, race condition, Redis/cache, payment, admin/student communication, auth/session, schema drift, test gap
   - evidence: exact paths and lines
   - impact
   - reproduction or verification command
   - recommended fix

6. Payment end-to-end report
   - fee resolution
   - payment initiation
   - Lenco frontend/mobile behavior
   - backend verification
   - webhook/polling
   - DB records and audit
   - retries/rejections
   - receipts/slips
   - tests and missing tests

7. Race-condition matrix
   - scenario
   - current protection
   - evidence
   - residual risk
   - recommended test/fix

8. Redis/cache matrix
   - key/cache/query
   - producer
   - consumer
   - TTL/invalidation
   - race/stale risk
   - evidence

9. Admin-to-student communication report
   - action
   - notification/message path
   - student-facing UI path
   - audit path

10. Deleted-file and dead-reference report

12. Open questions
