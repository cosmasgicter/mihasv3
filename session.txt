You are an agentic engineering team working on the MIHAS admissions system.

Mission:
Harden session management so students are not automatically logged out during the application process unless the session is truly unrecoverable. The system must preserve application progress, recover from transient auth/session failures, explain the actual issue to the user, and avoid defaulting to logout for recoverable errors.

Work with facts only:
- Do not assume the cause of logout.
- Prove each cause from code, logs, tests, network behavior, or reproducible steps.
- Treat comments and intended behavior as claims, not evidence.
- Every finding and fix must cite exact file paths and line numbers.
- Do not remove security controls. Harden recovery without weakening authentication, CSRF, ownership checks, or role permissions.

Scope:
- Frontend auth/session code in `apps/admissions/src/contexts`, `apps/admissions/src/hooks/auth`, `apps/admissions/src/services`, `apps/admissions/src/components/*Route*`, `apps/admissions/src/components/navigation`, `apps/admissions/src/pages/student/applicationWizard`, and related tests.
- Backend auth/session code in `backend/apps/accounts`, JWT/token handling, JTI blacklist, refresh/logout/session endpoints, middleware, CSRF handling, CORS/cookie settings, and related tests.
- API client retry/refresh code, cache invalidation, React Query session cache, localStorage/sessionStorage draft persistence, session monitor, route guards, and auth broadcast behavior.
- Application wizard persistence, autosave, draft resume/delete, file upload session recovery, payment step recovery, and dashboard polling.

Primary goals:
1. Identify every path that can redirect a student to `/auth/signin`, clear auth cache, clear CSRF/session state, dispatch `mihas:auth-expired`, or call logout.
2. Separate real auth failures from recoverable failures:
   - real auth failure: refresh token invalid/expired/blacklisted, explicit logout, revoked account, missing required auth cookie after retry, invalid role/permissions
   - recoverable failure: network outage, timeout, temporary API 5xx, generic 403 permission denial, CSRF token rotation, stale React Query cache, page refocus validation, BFCache restore, duplicate refresh race, stale local/session storage, upload retry failure, payment widget callback delay
3. Preserve user progress during recovery.
4. Auto-recover where safe:
   - retry session check
   - deduplicate token refresh
   - refresh CSRF token and retry state-changing requests
   - keep last known authenticated user during pending validation
   - pause autosave on transient auth uncertainty instead of clearing draft state
   - resume autosave after recovery
   - keep local draft snapshots until server persistence is confirmed
5. Tell the user what happened:
   - show “Reconnecting session” for transient recovery
   - show “We saved your progress locally” when online persistence is paused
   - show “Please sign in again” only after recovery fails with a real auth failure
   - never silently redirect away from the wizard while recovery is in progress

Required investigation:
- Build a complete logout/redirect map:
  - `signOut`
  - `configureApiClientAuthFailure`
  - `mihas:auth-expired`
  - `mihas:auth-redirect`
  - `authSignedOut`
  - `broadcastLogout`
  - route guards
  - session monitor
  - application wizard auth recovery
  - dashboard 403 handling
  - file upload auth recovery
  - autosave auth-expired handling
- For every path, record:
  - trigger
  - code location
  - whether it clears cache/storage
  - whether it preserves redirect target
  - whether it preserves drafts
  - whether it is recoverable
  - user-facing message

Frontend hardening requirements:
- Route guards must not redirect while session validation is pending and a cached user exists.
- Page refocus and BFCache session checks must preserve the last known user until verification completes.
- Generic `403` permission errors must not be treated as logout unless the backend explicitly says the token/session expired.
- API client must not call the global auth failure cascade for validation errors, ownership errors, payment errors, document errors, or generic permission denials.
- Concurrent `401` responses must share one refresh attempt.
- Session refresh success must update CSRF/session cache without invalidating unrelated application data unnecessarily.
- Failed refresh must be debounced so multiple failing requests do not dispatch repeated logout events.
- Wizard must preserve local draft before any auth recovery, and must not clear local draft on transient errors.
- Autosave must pause and explain when auth is uncertain; it must not mark the session expired permanently until recovery fails.
- File uploads must retry auth/session once, then preserve selected files and explain next steps.
- Payment flow must not lose the application state during token refresh, Lenco callback delay, verification polling, or page refocus.
- Student dashboard polling must not redirect to sign-in because one or more non-auth endpoints returned a generic 403.

Backend hardening requirements:
- Authentication failures must return clear machine-readable codes:
  - `AUTHENTICATION_REQUIRED`
  - `TOKEN_EXPIRED`
  - `TOKEN_BLACKLISTED`
  - `REFRESH_EXPIRED`
  - `CSRF_INVALID`
  - `INSUFFICIENT_PERMISSIONS`
- `401` must mean unauthenticated or unrecoverable auth after refresh.
- `403` must mean authenticated but not allowed, except where existing middleware explicitly converts expired JWT cases to `401`.
- Refresh endpoint must be idempotent enough for frontend retry/deduplication expectations.
- Logout endpoint must not be called by passive recovery paths.
- CSRF failures must be recoverable by a session/CSRF refresh when the auth cookies are still valid.
- JTI blacklist behavior must be tested so blacklisted refresh tokens produce a final sign-in-required state with a clear message.
- Cookie settings must be verified for production and local development:
  - `HttpOnly`
  - `Secure`
  - `SameSite`
  - domain/subdomain compatibility
  - path
  - expiry
  - refresh token rotation behavior

Persistence requirements:
- Application wizard form data must survive:
  - tab refocus
  - page reload
  - transient network outage
  - expired access token with successful refresh
  - CSRF retry
  - failed document upload
  - payment pending/verification retry
  - route guard recovery delay
- Local draft storage must not be deleted unless:
  - draft is submitted successfully and confirmed
  - user explicitly deletes draft
  - server confirms draft no longer exists and local state is stale beyond recovery
  - user explicitly signs out
- If server persistence fails, local draft must remain and the UI must show the exact recovery status.

User-facing diagnostics:
- Add or verify clear messages for:
  - “Reconnecting your session…”
  - “Your progress is saved on this device while we reconnect.”
  - “Connection issue. We will retry automatically.”
  - “Permission denied for this action, but you are still signed in.”
  - “Your session expired. We saved your progress. Please sign in again to continue.”
  - “Payment is still being verified. Do not start another payment unless prompted.”
- Avoid generic “Something went wrong” for auth/session failures.
- Avoid redirecting before the message can be displayed unless the user is already on a public route.

Race-condition checklist:
- Multiple simultaneous API calls returning `401`
- Multiple simultaneous refresh attempts
- Refresh success racing with route guard redirect
- Refresh failure racing with local draft autosave
- CSRF retry racing with refresh retry
- Page visibility revalidation racing with wizard autosave
- BFCache restore racing with session query
- Logout broadcast from another tab
- Explicit user logout versus transient recovery
- Upload retry while token refresh is in progress
- Payment verification while token refresh is in progress
- Dashboard polling while wizard mutation is in progress

Required implementation plan:
1. First produce a session failure map.
2. Then identify root causes ranked by likelihood and severity.
3. Implement fixes only after evidence is gathered.
4. Keep fixes narrowly scoped.
5. Add tests that fail on the current bug and pass after the fix.
6. Run focused tests and type-check.

Required tests:
- API client:
  - concurrent `401`s produce one refresh call
  - refresh success retries original request
  - refresh failure dispatches one auth-expired event only
  - generic `403 INSUFFICIENT_PERMISSIONS` does not log out
  - `CSRF_INVALID` refreshes CSRF and retries once
- Auth context/session listener:
  - pending validation keeps cached user
  - transient session fetch timeout keeps cached user and shows recovery
  - BFCache/pageshow validation does not null the user
  - explicit logout clears session and draft intent safely
- Route guards:
  - no redirect while cached user exists and session is validating
  - redirect only after recovery fails
- Wizard:
  - draft persists through auth recovery
  - autosave pauses on transient auth failure and resumes after recovery
  - selected upload files are not lost after auth retry failure
- Dashboard:
  - generic 403 from one endpoint does not navigate to sign-in
  - polling does not overwrite fresh wizard/payment updates
- Payment:
  - payment initiation survives token refresh
  - verification retry survives transient auth failure
  - pending payment message persists

Verification commands to run where applicable:

- frontend focused tests from `apps/admissions`
- backend auth/session tests from `backend`
- `bun run type-check` in `apps/admissions`
- backend unit/property tests for auth, sessions, CSRF, JTI blacklist, middleware, application draft persistence, and payments

Final acceptance criteria:
- The student is not logged out for transient network, CSRF, cache, generic permission, polling, upload, payment, or refocus failures.
- The student is only asked to sign in again after a true unrecoverable auth failure.
- Application progress is preserved before any redirect.
- Recovery attempts are deduplicated and bounded.
- The UI tells the student what is happening instead of silently redirecting.
- Tests prove the hardened behavior.
