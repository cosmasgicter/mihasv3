# MIHAS Infrastructure and Architecture Plan

Date: 2026-04-21
Scope: Full platform architecture review across `backend/`, `apps/admissions/`, `apps/jobs-ops/`, async workers, deployment/runtime assumptions, and domain exposure boundaries.

This version supersedes the earlier planning draft. It now records:
- what has already been fixed
- how each completed item improves the system
- what is still partially done or still open
- the latest end-to-end infra audit findings

Status legend:
- `DONE` = implemented in the repo
- `PARTIAL` = materially improved, but not yet complete
- `OPEN` = not yet implemented or only addressed by policy, not code

---

## 1. Executive Summary

The system is in a better state than the original architecture review, but it is not fully hardened yet.

What is now materially better:
- DRF is effectively the live authentication authority.
- Middleware auth is no longer part of the production auth path.
- Idempotency now behaves like command replay for same-actor same-method same-path same-body requests.
- Scaffold Jobs/Ops routes are disabled by default outside local development.
- Email dispatch is more durable because request flows persist `EmailQueue` rows before best-effort broker push.
- Contract parity tests no longer require a local Postgres instance for the narrow recorded-suite path.

What is still structurally weak:
- Redis semantics are now documented and formalized, but they remain intentionally mixed by policy.
- The generic outbox ledger now exists, but worker lifecycle hardening is still strongest on email and not yet uniform across every side effect.
- Schema governance is now explicit at the decision/process level, but still depends on SQL artifact discipline and live verification.
- `jobs-ops` is intentionally out of scope for the admissions frontend contract and should evolve under its own domain/runtime contract.
- Deployment documentation has been realigned with the current readiness contract and dependency-tier policy.

Bottom line:
- The architecture is no longer breaking for the original reasons at the same severity.
- The next failures are more likely to come from schema/process drift, incomplete outbox lifecycle parity, and local-vs-prod parity gaps.

## 2. Current State Verified In Code

### 2.1 Auth Boundary

Verified state:
- `JWTAuthenticationMiddleware` remains in code as a compatibility/test helper, but is no longer intended to be the production auth authority.
- DRF authentication classes are the active request auth path.
- `SessionView` uses optional DRF auth instead of relying on middleware-populated `request.user`.
- Remaining `authentication_classes = []` endpoints are concentrated in true public endpoints: login, register, refresh, password reset, health, meta, error reporting, payment webhook, and scaffold webhooks.

System impact:
- The original middleware-vs-DRF conflict has been materially reduced.
- Public bootstrap/session behavior is more predictable.
- The backend is closer to a single-source-of-truth auth model.

Residual risk:
- The repo still needs a formal ADR and a stricter inventory policy so new bypass endpoints do not creep back in.

### 2.2 Idempotency

Verified state:
- Idempotency now keys off `idempotency_key + actor_id + method + path` and compares `request_hash`.
- Same key + same payload replays.
- Same key + different payload returns `409 IDEMPOTENCY_CONFLICT`.
- Failed attempts are not cached as final results.
- Replay-safe JSON serialization is in place for DRF and JSON responses.

System impact:
- Duplicate client retries are much safer.
- Validation or permission failures do not poison the key forever.
- The previous blind key-cache behavior is no longer the active model.

Residual risk:
- The DB uniqueness constraint still does not include `request_hash`.
- Path identity is still literal `request.path`, not a canonicalized route template.
- No TTL cleanup or governance policy is implemented yet.

### 2.3 Domain Exposure

Verified state:
- Scaffold Jobs/Ops routes are gated by `ENABLE_JOBS_OPS_ROUTES`.
- Base settings now default this to `false`.
- Dev settings default it to `true`.

System impact:
- Production admissions traffic is no longer forced to share public blast radius with scaffold domains by default.
- This is the single biggest exposure hardening change outside auth.

Residual risk:
- If Jobs/Ops routes are explicitly enabled, scaffold webhook endpoints are still permissive and not production-grade.
- `jobs-ops` intentionally diverges from the admissions frontend contract, so it needs its own explicit runtime contract if it becomes production-significant.

### 2.4 Async Delivery

Verified state:
- Many email-producing code paths now call `dispatch_email(...)` after persisting `EmailQueue`.
- Shared durable helpers now also write a generic `outbox_events` ledger for email and notification side effects.
- Best-effort failures in communication flows log warnings instead of creating excessive stack-trace noise.
- A sweep task exists to redispatch stale pending emails.
- Email delivery now atomically claims rows before sending and reclaims stale `processing` rows back to `retrying` for re-dispatch.

System impact:
- A transient broker outage is less likely to drop intent entirely.
- Request success is less tightly coupled to broker reachability.

Residual risk:
- A generic outbox table now exists, but dispatch lifecycle hardening is still uneven.
- Claim safety exists for email delivery, but there is no dedicated persisted lease timestamp yet.
- Notifications are recorded in the outbox ledger, but they do not yet have the same worker-driven lifecycle as email.

### 2.5 Dependency Modes

Verified state:
- Readiness returns `200` when Postgres is healthy even if Redis is degraded.
- JTI blacklist reads fail closed after retry.
- Token rotation lock acquisition fails open if Redis is unavailable.
- Rate limiting fails open if the shared cache is unavailable.

System impact:
- The runtime is more available during Redis incidents.
- Security-sensitive paths are at least partially protected by fail-closed blacklist reads.

Residual risk:
- The dependency policy is intentionally mixed and now documented.
- The remaining risk is policy drift if auth, readiness, or rate-limit behavior changes without updating the ADR and runbooks.

### 2.6 Frontend Platform

Verified state:
- `apps/admissions` contains the mature cookie-auth, refresh, CSRF, timeout, and retry behavior.
- `apps/jobs-ops` uses a much thinner `credentials: include` API client with generic retry logic only.

System impact:
- Admissions remains the better-hardened frontend.
- Jobs/Ops can evolve independently without inheriting admissions-specific browser/runtime constraints.

Residual risk:
- The two frontends now need separate explicit contracts and ownership if they continue to diverge intentionally.

## 3. Work Already Completed

These are the notable remediations already present in the repo.

1. Auth authority was narrowed.
   Done:
   `backend/apps/common/middleware.py` keeps middleware auth semantics only as a compatibility/test helper instead of a production authority.
   Improves:
   Reduces the risk of middleware and DRF fighting over `request.user`.

2. Session bootstrap path was hardened.
   Done:
   `backend/apps/accounts/views.py` uses `OptionalJWTCookieAuthentication` for session discovery.
   Improves:
   Public pages can probe auth state without turning absence of auth into a backend-side failure.

3. Idempotency logic was redesigned from blind key replay to command identity.
   Done:
   `backend/apps/common/idempotency.py` now checks actor, method, path, and body hash, replays safe JSON payloads, and avoids poisoning keys on failed requests.
   Improves:
   Duplicate submissions are safer and more predictable.

4. Scaffold Jobs/Ops public exposure was tightened.
   Done:
   `backend/config/settings/base.py` disables these routes by default; `backend/config/settings/dev.py` re-enables them for local development.
   Improves:
   Production surface area is smaller by default.

5. Email dispatch became more durable.
   Done:
   `dispatch_email(...)` plus stale-pending sweep behavior were normalized across multiple task and service modules.
   Improves:
   Persisted delivery intent survives transient broker failures more often.

6. A generic outbox ledger was added.
   Done:
   `backend/apps/common/outbox.py`, `backend/apps/common/models.py`, and `backend/scripts/add_outbox_events.sql` now persist normalized `outbox_events` rows for email and notification side effects.
   Improves:
   Gives the platform one durable audit and extension point for side effects beyond email-specific delivery rows.

7. Communication flow noise was reduced.
   Done:
   `backend/apps/common/communication_service.py` treats best-effort failures as warnings rather than stack-trace storms.
   Improves:
   Operators get cleaner signal during degraded integrations or template lookup failures.

8. Contract parity became locally reproducible for the narrow recorded suite.
   Done:
   `backend/tests/contract/conftest.py` bootstraps a minimal SQLite schema for contract recordings.
   Improves:
   Architecture verification no longer depends on a live local Postgres instance for that limited suite.

## 4. Architectural Principles

These principles still govern the recommended target state.

1. One authority per concern.
2. Stable transport semantics.
3. Domain isolation before service extraction.
4. Durable intent before async execution.
5. Explicit dependency tiers.
6. Explicit frontend contracts per product boundary.
7. Schema discipline is non-negotiable.

## 5. Target Architecture

### 5.1 Platform Layers

Layer A: Edge and delivery
- Vercel-hosted frontend(s)
- Koyeb-hosted backend ASGI service
- DNS, TLS, origin policy, proxy rules

Layer B: Platform core
- auth
- sessions
- permissions/RBAC
- notifications
- audit
- settings
- error reporting

Layer C: Business domains
- admissions
- documents
- payments
- catalog
- jobs/ops
- outreach
- automation
- analytics

Layer D: Async orchestration
- outbox tables
- Celery workers
- periodic jobs
- queue routing

Layer E: Persistence and state
- Postgres
- Redis
- object storage

### 5.2 Auth Target State

Target:
- DRF is the only runtime auth authority.
- Middleware never authenticates users in production.
- Public endpoints are explicitly public.
- Optional auth is explicit.
- Protected endpoints rely on DRF auth plus permission classes.

HTTP contract:
- `401` = unauthenticated/expired/invalid token
- `403` = authenticated but forbidden
- `409` = semantic conflict
- `400` or `422` = validation, chosen consistently

### 5.3 Idempotency Target State

Target identity:
- idempotency key
- actor id
- method
- normalized route
- request hash

Behavior:
- identical duplicate returns canonical result
- same key with different body returns conflict
- only explicit command classes are idempotent
- TTL and cleanup are documented

### 5.4 Domain Target State

Core platform:
- accounts
- auth
- sessions
- permissions
- audit
- notifications

Admissions product:
- applications
- catalog
- documents
- payments

Jobs/Ops product:
- jobs
- outreach
- automation
- analytics
- integrations

Rule:
- immature domains must not expand admissions production risk.

### 5.5 Async Target State

Flow:
1. domain write transaction
2. outbox row creation in same transaction
3. dispatcher/worker claims row
4. delivery retry policy executes
5. terminal status is persisted

### 5.6 Dependency Tier Target State

Tier 1:
- Postgres
- backend runtime

Tier 2:
- Redis for degraded-safe coordination and caching

Tier 3:
- no security-critical behavior should depend on best-effort Redis semantics without explicit policy

### 5.7 Frontend Platform Target State

Shared browser platform layer:
- API client core
- auth/session state
- timeout/retry policy
- CSRF handling where applicable
- request tracing
- route protection
- consistent envelope parsing

## 9. Fresh End-To-End Infra Audit

This section records the latest audit findings after the implemented hardening.

### Finding 1: Deployment docs were previously stale, and are now corrected

Evidence:
- `backend/apps/common/health.py` returns `200` with `redis: degraded` when Postgres is healthy and Redis is down.
- `backend/DEPLOY.md` now documents that `/health/ready/` returns `503` for Postgres failure but remains `200` with `redis: degraded` when only Redis is down.

Risk:
- The immediate doc/code mismatch is resolved.
- The remaining risk is policy drift if dependency-tier decisions change without keeping the runbook aligned.

Recommendation:
- Keep `DEPLOY.md`, readiness behavior, and ADR-007 aligned as a single contract.

### Finding 2: Redis still has mixed security and availability semantics

Evidence:
- Token rotation lock acquisition fails open if Redis is unavailable.
- JTI blacklist reads fail closed after retry.
- Readiness stays `200` under Redis degradation.
- Rate limiting fails open on cache failure.

Risk:
- Security guarantees are mode-dependent and not obvious.
- A Redis outage can preserve uptime while changing auth behavior in subtle ways.

Recommendation:
- Keep ADR-007, runbooks, and code behavior aligned.
- If policy changes later, choose one explicit stance for auth under Redis outage:
  - security-first: fail refresh/rotation when Redis is down
  - availability-first: accept weaker guarantees and expose that mode clearly

### Finding 3: Generic outbox recording now exists, but lifecycle hardening is still email-first

Evidence:
- `outbox_events` now records normalized email and notification side effects.
- `dispatch_email(...)` is best-effort.
- `process_pending_emails_task` sweeps rows with `status='pending'` and reclaims stale `processing` rows.
- `send_email_task` now atomically moves rows into `processing` before delivery.

Risk:
- Immediate duplicate queued tasks are now harmless for email delivery.
- The remaining risk is lifecycle asymmetry: email has claim/retry behavior, while other outbox-recorded side effects do not yet have equivalent worker orchestration.

Recommendation:
- Add a persisted lease/claim timestamp when the schema allows it.
- Extend the same lifecycle model beyond email into the broader `outbox_events` policy.

### Finding 4: Idempotency is improved but not fully normalized

Evidence:
- The decorator uses actor, method, path, and request hash.
- The DB uniqueness constraint is still on `idempotency_key + actor_id + method + path`.
- The route identity is still literal path text.

Risk:
- Command identity is stronger than before but not yet canonical enough for long-term guarantees.
- Cleanup/retention is undefined.

Recommendation:
- Add a canonical route field or normalized endpoint identity.
- Add retention/TTL policy and cleanup job.
- Publish an endpoint eligibility matrix.

### Finding 5: Schema governance remains the biggest unresolved structural risk

Evidence:
- Large parts of the backend still use `managed = False`.
- There is no repo-visible schema promotion/runbook replacing Django migrations as the source of truth.

Risk:
- Code can pass tests while drifting from production schema reality.
- Changes become dependent on tribal knowledge.

Recommendation:
- Keep ADR-004 backed by enforceable schema checks in CI.
- Add schema snapshots and compatibility checks in CI.
- Document whether SQL migrations live elsewhere or need to be introduced here.

### Finding 6: Frontend contract separation is intentional

Evidence:
- `apps/admissions` has a mature cookie-auth, refresh, CSRF, timeout, and retry stack.
- `apps/jobs-ops` uses a simpler API client with generic retries.
- `shared/PLATFORM_CONTRACT.md` now explicitly scopes itself to admissions only.

Risk:
- If jobs-ops becomes production-significant, it will need its own explicit contract instead of silently drifting.

Recommendation:
- Define a separate jobs-ops contract when that product/runtime boundary hardens.

### Finding 7: Local infra parity is still weak

Evidence:
- `backend/docker-compose.yml` provisions Redis/web/celery only.
- Local Postgres parity still depends on external Neon or ad hoc setup.
- SQLite harness now exists for contract tests, but that is not full environment parity.

Risk:
- End-to-end backend behavior is harder to reproduce locally.
- CI and local debugging can diverge from production characteristics.

Recommendation:
- Add an optional local Postgres service or a documented local parity profile.
- Separate "fast local" and "prod-like local" workflows.

### Finding 8: Scaffold webhook surfaces remain unsafe if re-exposed

Evidence:
- Integration/email webhook scaffold views are `AllowAny` and unauthenticated.
- They are route-gated by default now, but they are still placeholders if re-enabled.

Risk:
- A future route toggle could expose non-production webhook handlers without signature validation or business rules.

Recommendation:
- Mark scaffold webhook endpoints as internal-preview only.
- Require signature validation before any production exposure.

---

## 10. Highest-Value Next Improvements

1. Publish an idempotent endpoint matrix and add retention/cleanup policy.
2. Add a local Postgres parity profile for reproducible infra debugging.
3. Document domain maturity and keep scaffold webhooks non-production until signed/validated.
4. Add a persisted lease/claim timestamp for email delivery when the schema workflow is in place.
5. Extend worker lifecycle parity from email into notifications and other outbox-recorded side effects.
6. Add dedicated Redis-only monitoring if degraded Redis should page.

---

## 11. Final Recommendation

The architecture is no longer primarily blocked by the original auth-layer conflict. That part is materially better.

The next stage is different:
- reduce dependency-mode ambiguity
- finish outbox hardening
- make schema ownership explicit
- keep frontend contracts explicit per product boundary

If the next round of work focuses on those four areas, the system will become easier to operate, safer under degradation, and less likely to regress into hidden architectural ambiguity.
