## Read This First

This document is a handover for another advanced AI or engineer. It is not gospel truth. Treat it as a map of the intended architecture and the current implementation foundation, then verify every claim against the repo, the database, and tests before extending it.

Rules for the next AI:

- The codebase is the source of truth. Use `rg`, file reads, tests, and database introspection before acting.
- Keep the platform canonical-ID first. Legacy string fields exist for display snapshots and backward compatibility only.
- Do not infer cross-school access from role alone. Super admin is global. School staff must use memberships and explicit grants.
- Do not add arbitrary form builders or DOCX/PDF merge engines in V1. Templates are safe sections, tokens, and uploaded assets.
- Do not assume migrations have been applied in production. Confirm the migration history prerequisite first.
- Do not remove legacy string fields yet. They are needed to preserve old applications and generated documents.
- Do not trust counts, exports, search, document URLs, or payment endpoints until scoped-access tests prove they cannot leak other schools' data.

## Product Goal

Convert the original MIHAS/KATC admissions system into a Beanola-owned multi-school admissions platform.

Beanola is the platform owner. MIHAS, KATC, and future schools are client institutions. Students apply program-first:

1. The student chooses a canonical program and intake.
2. The backend assigns an eligible school-specific offering.
3. The student sees the assigned school, fee, and required documents before payment/submission.
4. Beanola collects the payment centrally.
5. Payments are tagged by institution, canonical program, offering, and intake for settlement/reporting.
6. Official documents are generated by the backend using the assigned school's brand, templates, assets, and version metadata.

The platform supports:

- Shared Beanola portal: Beanola brand before assignment.
- School white-label domains: school brand and school-filtered offerings before assignment.
- One core V1 application form.
- Per-school/program required documents, official templates, assets, emails, contacts, branding, signatures, fees, and copy.

## Database Plan

The production schema source of truth remains SQL files under `backend/scripts/`. The migration is additive and intended to be idempotent.

### New Shared Program Table

`canonical_programs`

Purpose: shared cross-school program definitions. This is the canonical program identity students choose.

Key fields:

- `id`
- `name`
- `code`
- `description`
- `duration_months`
- `regulatory_body`
- `is_active`
- `metadata`
- timestamps

### Existing `programs` Become Offerings

Existing `programs` are school-specific offerings.

Added fields:

- `canonical_program_id`
- `assignment_priority`
- `offering_status`
- `assignment_rules`

Expected meaning:

- `canonical_program_id`: points to the shared program definition.
- `institution_id`: existing school owner of this offering.
- `offering_status`: active offerings can receive new applications; archived/inactive offerings remain readable.
- `assignment_priority`: lower priority wins during assignment unless intake-specific priority overrides it.
- `assignment_rules`: JSON rules for residency/country/nationality eligibility.

### Program Intake Assignment Fields

Added to `program_intakes`:

- `is_active`
- `assignment_priority`
- `residency_rules`

Use `ProgramIntake` to control offering availability per intake, intake capacity, and intake-specific routing priority.

### Application Canonical IDs

Added nullable IDs to `applications`:

- `institution_id`
- `program_id`
- `program_offering_id`
- `intake_id`

Important:

- These are source-of-truth references for new flows.
- Existing string fields such as `institution`, `program`, and `intake` must remain as immutable display snapshots/backward compatibility.
- Do not build new business logic from strings except migration/backfill or legacy fallback code.

### Tenant Configuration Tables

New tables:

- `institution_assets`
- `institution_document_templates`
- `institution_required_documents`
- `institution_domains`
- `user_institution_memberships`
- `access_grants`

Purpose summary:

- `institution_assets`: versioned logos, signatures, seals. Store R2/S3 key, MIME, checksum, version, metadata.
- `institution_document_templates`: versioned safe sections/tokens for official PDFs.
- `institution_required_documents`: configurable document requirements per institution, offering, canonical program, or default.
- `institution_domains`: white-label hostnames.
- `user_institution_memberships`: staff access to institutions.
- `access_grants`: explicit extra access at institution, program offering, or application level.

### Migration Backfill

The current migration attempts:

- Create canonical programs from existing programs by code.
- Link programs to canonical programs by code.
- Fill institution slug, brand name, support email, admissions email.
- Resolve existing applications to institution/program offering/canonical program/intake IDs by case-insensitive string matching.

Backfill is best effort. It must be audited. Name/code collisions and historical spelling variants can leave null IDs.

### Production Migration Procedure

Do not skip these steps.

1. Confirm the migration-history prerequisite has been applied.

   Known blocker in this checkout:

   - `apply_sql_migrations --dry-run` was blocked until `2026_05_22_migration_history_extend.sql` is applied.

2. Take a database backup.

3. Run the migration dry run using the repo's existing SQL migration command.

4. Apply the SQL migration on a staging copy first.

5. Run post-migration SQL validation:

   ```sql
   select count(*) as canonical_programs from canonical_programs;
   select count(*) as programs_without_canonical from programs where canonical_program_id is null and coalesce(is_active, true) = true;
   select count(*) as applications_without_institution_id from applications where institution_id is null;
   select count(*) as applications_without_program_id from applications where program_id is null;
   select count(*) as applications_without_offering_id from applications where program_offering_id is null;
   select count(*) as applications_without_intake_id from applications where intake_id is null;
   select hostname, count(*) from institution_domains group by hostname having count(*) > 1;
   select slug, count(*) from institutions where slug is not null group by slug having count(*) > 1;
   ```

6. Validate not-valid foreign keys after resolving backfill issues, not before.

7. Create a manual exception report for legacy applications that cannot be linked by IDs.

8. Verify read paths can still display legacy applications with null IDs.

9. Verify new application creation writes all four new IDs.

10. Verify official documents generated before migration remain readable and unchanged.

## Backend Architecture

### Assignment Service

Service: `OfferingAssignmentService`

Inputs:

- canonical program ID
- intake ID
- optional country
- optional nationality
- optional institution ID for white-label filtering

Expected algorithm:

1. Load active canonical program and active intake.
2. Find active programs where:
   - `canonical_program_id` matches
   - offering is active
   - offering is available for the intake
   - program intake is active or legacy null-active
3. If white-label context supplied an institution ID, filter to that institution.
4. Apply offering assignment rules.
5. Apply program-intake residency rules.
6. Check capacity.
7. Sort by intake priority, then offering priority, then code/id tie-break.
8. Return assigned offering, institution, canonical program, intake, and required documents.

Needed improvements:

- Revalidate assignment at submission, not only draft creation.
- Define and test waitlist behavior when capacity changes after draft creation.
- Add deterministic tests for ties.
- Add tests for archived offering read/new-application deny.
- Add tests for country/nationality allow/block rules.
- Add audit event for assignment decisions.

### Institution Context

Service: `InstitutionContextService`

Purpose:

- Resolve shared Beanola portal vs white-label school host.
- Return brand context.
- Filter catalog choices for white-label domains.

Current behavior:

- If `institution_domains.hostname` matches active host and institution is active, return white-label context.
- Otherwise return shared Beanola brand.

Needed improvements:

- Add domain verification lifecycle.
- Add tests for uppercase host, ports, inactive domains, inactive institutions, and collisions.
- Add cache with explicit invalidation on domain changes.

### Access Scope

Service: `AccessScopeService`

Current model:

- Super admin gets all access.
- Regular users get:
  - institution IDs from active memberships
  - institution/program/application IDs from active non-expired grants
- Query filters exist for applications, payments, and documents.

Important caveat:

- There is a test-settings-only compatibility path for legacy admin unit fixtures with no memberships. Do not copy that behavior into production. Production regular admins must be scoped by membership/grants.

Already scoped:

- Admin application list/review path.
- Application exports.
- Payments list/receipt/verify.
- Settlement summary.
- Documents retrieval/download/delete/admin upload.
- Admin dashboard aggregates/activity/needs-attention counts.
- Admin user listing.
- Tenant admin reads.

Still needs hardening:

- Analytics endpoints.
- Any future global search/count endpoints.
- Any non-obvious document signed URL paths.
- Risk views and operational reports if exposed to regular school staff.

Security rule:

Non-super-admin users must not infer other schools' data through counts, empty states, search timing, exports, document URLs, payment receipt URLs, settlement summaries, or error messages.

### Document Templates and Official PDFs

Service: `DocumentTemplateService`

Renderer: `backend/apps/applications/tasks/pdf_generation.py`

Official document types:

- application slip
- acceptance letter
- conditional offer
- finance receipt
- payment receipt

Current behavior:

- Official PDFs use assigned school context where available.
- Active latest logo/signature assets are included when drawable.
- Template sections are token-rendered with HTML escaping.
- Generated document metadata is stored in `ApplicationDocument.verification_notes` under `official_document`.

Rules:

- Backend-generated documents are canonical.
- Frontend PDFs are preview/download helpers only.
- Do not allow arbitrary uploaded DOCX/PDF merge engines in V1.
- Logos/signatures must be versioned and never overwrite already generated documents.
- Always snapshot template version and asset IDs into generated document metadata.

Needed improvements:

- Add PDF visual regression or content extraction tests.
- Add template token allowlist tests.
- Add tests for malicious template tokens and unexpected nested JSON.
- Store document render failures in auditable form.
- Add school contact/footer customization.
- Ensure SVG assets are either safely rendered or deliberately unsupported with clear UI copy.

## Canonical-Only Standard

Use this standard for every new change.

Canonical program:

- Source of truth: `canonical_programs.id`.
- Display snapshot: `applications.program`.
- Do not match by program name in runtime logic except legacy fallback.

School offering:

- Source of truth: `programs.id` as `program_offering_id`.
- Must be linked to `canonical_program_id`.
- Must keep `institution_id`.

Institution:

- Source of truth: `institutions.id`.
- Slug/code are identifiers but not FK replacements.
- Display snapshot: `applications.institution`.

Intake:

- Source of truth: `intakes.id`.
- Display snapshot: `applications.intake`.

Application uniqueness:

- One active application per student identity per canonical program + intake.
- This applies even if multiple schools offer that canonical program.
- Identity logic must account for same user applying for another person only when NRC/passport differ.

Payment reporting:

- Use payment metadata snapshots for reporting.
- Do not recalculate historical settlement labels only from current institution/program names.

Documents:

- Use generated document metadata for template/asset versions.
- Do not regenerate historical documents silently after template/asset changes.

## Final Guidance For The Next AI

The foundation is substantial, but the job is not simply "add features." The main risk is cross-tenant leakage or accidentally returning to string/name based logic. Your first priority should be proving the canonical-ID model and access-scope model under real database conditions.

Before adding UI polish:

1. Verify database migration readiness.
2. Verify canonical IDs on new application creation.
3. Verify scoped access on every admin, payment, document, dashboard, export, and analytics path.
4. Verify official generated documents snapshot template and asset versions.
5. Verify white-label context does not leak other schools.

Only then improve the UI.
